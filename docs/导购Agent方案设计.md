# 导购 Agent 方案设计

## 1. 目标

基于当前 `MallPilot` 已有商品数据，设计一个可落地的导购 Agent，支持以下核心能力：

1. 基于商品知识库进行商品理解、推荐与问答。
2. 支持文本、图片、图片加文本的多模态商品召回。
3. 支持加入购物车、直接下单、查询订单、取消订单、申请退货等交易闭环。
4. 后续可以扩展到用户画像、个性化推荐、售后问答与运营分析。

## 2. 现有数据现状

当前仓库内已有一套非常适合做导购 Agent 的基础数据：

1. 数据位置：`data/ecommerce_agent_dataset`
2. 品类数量：4 类
3. 商品数量：100 个 `json`
4. 商品图片数量：100 张
5. 每个商品同时具备：
   - 基础结构化信息：`product_id`、`title`、`brand`、`category`、`sub_category`、`base_price`
   - SKU 信息：`sku_id`、规格属性、SKU 价格
   - 图片路径：`image_path`
   - RAG 知识：`marketing_description`、`official_faq`、`user_reviews`

这意味着你现在已经具备了一个导购 Agent 最重要的三类原料：

1. 商品结构化数据
2. 商品长文本知识
3. 商品图片数据

## 3. 当前数据风险

在方案落地前，建议优先处理以下问题，否则会直接影响检索质量和 Agent 可用性：

1. 当前环境下 `100/100` 商品 `json` 无法直接被 `ConvertFrom-Json` 解析。
2. 多个商品字段在终端读取时存在明显乱码，说明存在编码不统一或文本已被错误转码的问题。
3. 如果编码问题不先解决，Embedding、关键词召回、FAQ 切分、评论摘要、多模态对齐都会明显失真。

建议先做一层离线数据清洗：

1. 统一源文件编码为 `UTF-8`
2. 做 JSON 结构校验
3. 补齐或修复异常引号、非法字符、截断文本
4. 对 `title`、`brand`、`category`、`sku.properties` 做标准化
5. 给每个商品生成可检索的规范化文档

### 4.1 这些项目的共同模式

综合这 5 个项目，导购 Agent 大多都会采用下面几种共识设计：

1. 把“商品理解”和“交易执行”拆开，不让大模型直接裸调用底层订单系统。
2. 检索不是只有一个向量库，而是“结构化过滤 + 语义检索 + 规则重排”的混合召回。
3. 购物动作必须工具化，并且要做参数校验、库存校验、价格确认、二次确认。
4. 用户问题往往不是单轮问答，而是“需求澄清 -> 商品筛选 -> 比较推荐 -> 加购/下单”。
5. 售后场景不是普通 FAQ，而是强状态任务，需要订单状态、退货政策、时间窗口、物流信息共同参与。
6. 多模态能力不是锦上添花，而是服饰、美妆、家居、食品等类目提升体验的关键能力。

### 4.2 对 MallPilot 最有价值的近期启发

结合这 5 个仓库，最值得直接吸收的不是“界面长什么样”，而是下面几条实现经验：

1. `BuyPilot-AI` 的价值在于它把“约束满足率、召回率、多轮一致性、证据覆盖”明确当成导购 Agent 的核心指标，这和 MallPilot 非常契合。
2. `BuyPilot-AI` 还明确强调运行时使用 `PostgreSQL + pgvector`，并把混合检索、证据绑定、评测与可观测性做成默认能力，这和你当前想做的“SQL 过滤 -> pgvector 召回 -> rerank”路线一致。
3. `AuraCart` 的价值在于它把导购流程拆成显式节点，而不是一个黑盒 Agent，这很适合 MallPilot 第一版做成稳定、可调试的工作流。
4. `AuraCart` 还直接采用“结构化过滤 + 语义搜索 + 关键词搜索 + 融合重排”的思路，非常接近你当前已经确认的检索策略。
5. `ai-agent` 的价值在于它强调“快车道 + 主链路”双层路由，让 FAQ、单步工具调用和复杂 ReAct 分流执行，可以明显降低简单请求的成本。
6. `ai-agent` 还把商城商品、购物车、订单能力通过独立 `mall-mcp` 暴露，并对 `create_order` 类高风险工具设置确认门禁，这对下单、取消、退货场景非常重要。
7. `SmartShop` 的价值在于它把“商品查询”和“订单查询”拆成并发子任务，通过 Router 做汇总，这说明订单域完全可以作为独立子 Agent 处理。
8. `RAGent` 的价值在于它把导购主流程清晰拆成 `Master Agent + search / compare / scene / cart / order / product_inquiry` 六类子 Agent，这和我们前面讨论的“主 Agent + subagent”方向高度一致。
9. `RAGent` 还把“混合检索 + 状态机路由 + 购物车/下单闭环 + 多模态输入”放在一个统一编排里，说明导购系统最终不是单纯的 RAG 问答，而是一个任务型状态系统。

### 4.3 这 5 个项目对 MallPilot 的具体借鉴分工

如果只保留最有价值的吸收方向，我建议这样分：

1. 从 `BuyPilot-AI` 借鉴：评测指标体系、证据绑定、可解释推荐、`Postgres + pgvector` 运行时约束。
2. 从 `AuraCart` 借鉴：`LangGraph` 节点式工作流、SSE 事件协议、`pgvector + zhparser` 的双路检索思路。
3. 从 `ai-agent` 借鉴：小模型 JSON 路由快车道、MCP 商城边界、安全门禁、多层记忆设计。
4. 从 `SmartShop` 借鉴：商品域和订单域拆分、并发子任务、MCP 微服务化、Router 编排。
5. 从 `RAGent` 借鉴：主 Agent + 多 subagent、状态机路由、购物车与下单闭环、多模态输入到购物动作的完整链路。

## 5. 面向 MallPilot 的推荐架构

推荐把系统拆成五层：

1. 数据层
2. 索引层
3. Agent 编排层
4. Commerce 工具层
5. 安全与评测层

### 5.1 数据层

建议把当前商品数据拆成四类对象：

#### 5.1.1 商品主表

保留：

1. `product_id`
2. `title`
3. `brand`
4. `category`
5. `sub_category`
6. `base_price`
7. `image_path`

#### 5.1.2 SKU 表

保留：

1. `sku_id`
2. `product_id`
3. `properties`
4. `price`
5. `stock`
6. `status`

说明：

1. 你当前数据里已经有 SKU 属性与价格。
2. 建议后续补 `stock` 和 `status`，否则 Agent 只能推荐，不能可靠成交。

#### 5.1.3 商品知识表

将 `rag_knowledge` 拆成独立知识块：

1. `marketing_description`
2. `official_faq.question`
3. `official_faq.answer`
4. `user_reviews.content`
5. `user_reviews.rating`

这样可以在召回时区分“官方知识”和“用户口碑”，避免混淆。

#### 5.1.4 商品视觉表

保留：

1. 商品主图
2. 后续补充图
3. 图片 Embedding
4. 图片标签
5. 颜色、风格、材质等视觉属性

### 5.2 索引层

这里不建议只做一个大而全的 RAG 库，建议做“三路索引”：

#### 5.2.1 结构化检索索引

适合处理：

1. 品类过滤
2. 品牌过滤
3. 价格区间过滤
4. SKU 规格过滤
5. 库存过滤

建议使用关系型数据库或支持过滤的检索引擎。

#### 5.2.2 文本向量索引

向量化对象建议包括：

1. 商品标题
2. 商品卖点摘要
3. FAQ 问答块
4. 用户评论摘要

建议采用分块策略：

1. 商品基础信息一个 chunk
2. 营销描述按语义切分多个 chunk
3. 每条 FAQ 独立一个 chunk
4. 评论先做摘要再入库，避免噪音太大

#### 5.2.3 图片向量索引

图片向量库建议支持：

1. 以图搜商品
2. 图文联合召回
3. 风格相似商品查找

典型场景：

1. 用户上传一张连衣裙图片，问“帮我找类似款，预算 300 以内”
2. 用户上传口红图片，问“找相近色号”
3. 用户上传耳机图片，问“有没有类似外观但更便宜的”

### 5.3 检索策略

推荐采用“混合召回 + 重排”：

1. 第一步：从用户问题中抽取品类、品牌、预算、颜色、规格、用途
2. 第二步：结构化过滤缩小候选集
3. 第三步：文本向量召回相关商品知识
4. 第四步：如果有图片输入，再走图片向量召回
5. 第五步：做 cross-encoder 或 LLM rerank
6. 第六步：输出 TopN 商品并生成推荐理由

这样做比“直接把所有商品扔进一个向量库”更稳。

建议把这一段进一步固化为标准链路：

1. `SQL 硬过滤`
2. `pgvector 向量召回`
3. `gte rerank`

为什么要按这个顺序：

1. `价格`、`品牌`、`肤质`、`尺码`、`库存状态` 这类条件属于刚性约束，不是“尽量满足”，而是“必须满足”。
2. 向量检索擅长做语义相似匹配，但不适合承担严格业务约束校验。
3. 先用 SQL 把候选集限定在安全子集内，才能保证后续召回和排序都在可控范围内进行。
4. 向量召回更适合解决“这些合格商品里，哪些最像用户真正想要的”。
5. rerank 更适合解决“这些相关商品里，最终谁应该排在最前面”。

如果顺序反过来，即“先向量召回，再做硬过滤”，会出现下面的问题：

1. 会先召回大量本来就不应该进入候选集的商品，例如超预算、错品牌、不适合敏感肌、无库存商品。
2. 大量 TopK 名额会被无效候选占用，真正满足约束的商品反而可能进不了后续排序阶段。
3. 检索和重排资源会被浪费在无效商品上，整体成本更高。
4. 最终推荐结果会变得不稳定，容易出现“用户明确设了条件，但返回结果仍然不准”的问题。
5. 在美妆、服饰、数码这类规格敏感场景里，结果不可控会直接伤害用户信任。

例如：

1. 用户说：“帮我找 300 元以内、适合敏感肌、最好是雅漾的修护面霜”
2. 正确做法是先用 SQL 过滤 `price <= 300`、`brand = 雅漾`、`适合敏感肌` 这类硬条件
3. 再在这个小候选集里做向量召回，理解“修护”“维稳”“屏障修复”这些语义需求
4. 最后通过 rerank 把最符合当前购买意图的商品排在前面

因此，在导购 Agent 里建议固定一条原则：

1. 刚性业务约束先走结构化过滤
2. 语义相关性再走向量召回
3. 最终结果由 rerank 做精排

## 6. Agent 角色设计

建议不要只做一个大而全的单 Agent，推荐最少拆成下面四个角色：

### 6.1 导购主 Agent

负责：

1. 理解用户需求
2. 决定走推荐、问答、加购还是售后流程
3. 组织多轮对话
4. 汇总最终答复

### 6.2 商品检索 Agent

负责：

1. 结构化过滤
2. 文本检索
3. 图片检索
4. 商品比较
5. 推荐理由生成

### 6.3 交易执行 Agent

负责：

1. 加入购物车
2. 更新购物车
3. 创建订单
4. 支付前确认
5. 查询订单状态

关键原则：

1. 交易执行必须走工具调用。
2. 下单前必须把商品、SKU、数量、价格、地址、优惠、运费再次展示给用户确认。

### 6.4 售后 Agent

负责：

1. 取消订单
2. 申请退货
3. 查询退货政策
4. 查询退款状态
5. 必要时转人工

更合适的关系不是“四个完全同级 Agent”，而是：

1. `导购主 Agent` 作为编排者
2. `商品检索 Agent`、`交易执行 Agent`、`售后 Agent` 作为按需调用的 subagent

这样更符合 `RAGent` 这类近期项目的主 Agent + 多子 Agent 设计，也更容易做权限控制。

## 8. 关键交互流程

### 8.1 导购推荐流程

1. 用户表达需求
2. Agent 抽取约束条件
3. 如条件不足，先澄清
4. 检索候选商品
5. 输出 3 到 5 个推荐项
6. 给出推荐理由、优缺点、适用人群、价格差异
7. 引导用户比较、加购或继续缩小范围

### 8.2 多模态搜商品流程

1. 用户上传图片
2. 视觉模型提取图片特征
3. 识别品类、颜色、风格、材质、品牌线索
4. 走图片召回
5. 和文本条件做联合过滤
6. 输出相似商品与替代方案

### 8.3 加购与下单流程

1. 锁定商品
2. 明确 SKU
3. 明确数量
4. 校验库存与价格
5. 加入购物车或生成订单预览
6. 用户确认
7. 创建订单

### 8.4 取消订单流程

1. 获取订单状态
2. 判断是否允许取消
3. 告知用户取消后影响
4. 用户确认
5. 执行取消

### 8.5 退货流程

1. 获取订单与签收状态
2. 查询退货窗口和规则
3. 收集退货原因
4. 校验商品是否符合退货条件
5. 创建退货申请
6. 返回退货单号与后续步骤

## 9. 多模态导购建议

你的数据里已经有商品图，后续非常适合把多模态导购做成强能力，而不仅仅是一个附加功能。

### 9.1 推荐支持的输入形式

1. 纯文本
2. 单张图片
3. 图片加文本
4. 历史订单加文本

### 9.2 推荐支持的输出形式

1. 商品卡片
2. 商品对比表
3. 适用人群总结
4. 评论摘要
5. 可直接点击的加购、下单动作

### 9.3 特别适合多模态的类目

1. 服饰运动
2. 美妆护肤
3. 数码电子中的外观偏好型商品
4. 食品生活中的包装、规格、礼盒类商品

## 10. 推荐系统与 RAG 如何结合

不建议把“推荐系统”和“RAG”视为同一件事。

推荐做法是：

1. RAG 负责“理解商品”和“回答问题”
2. 推荐模块负责“挑选更可能成交的商品”
3. Agent 负责“在当前上下文里组织决策”

一个更合理的链路是：

1. 先用规则和向量召回商品
2. 再用推荐排序融合：
   - 用户偏好
   - 价格敏感度
   - 评论口碑
   - 品牌偏好
   - 历史行为
3. 最后由 Agent 生成自然语言推荐理由

## 11. 安全与风控建议

购物 Agent 比普通问答 Agent 更需要约束，建议至少加上这些规则：

1. 下单、取消、退货必须是工具调用，不允许模型自行编造结果。
2. 任何涉及金额变化的操作，都必须二次确认。
3. 必须把实时价格、库存、优惠、运费作为工具返回值展示给用户。
4. 当用户条件不明确时，Agent 可以推荐，但不能默认下单。
5. 涉及地址、支付、退款时，需要权限校验和审计日志。
6. 当订单状态不满足取消或退货条件时，必须明确拒绝并解释原因。

## 13. 一版推荐的技术实现思路

3. 图片 Embedding：多模态 Embedding 模型
4. Agent 编排：工作流式 Agent
5. 工具协议：参考这 5 个仓库里的工作流、MCP、子 Agent 分工方式设计
6. 前端动作：商品卡片、加购按钮、下单确认弹层、售后申请入口

## 14. 最适合你当前数据集的一版方案

结合你当前只有商品数据、还没有完整交易系统的现实情况，我最建议的落地路径是：

1. 先把商品知识库和多模态检索做扎实。
2. 再把购物车做成第一批真实工具。
3. 订单、取消订单、退货先做成模拟工具或状态机工具。
4. 等真实交易后端稳定后，再替换成正式 commerce API。

原因很直接：

1. 你现在最强的资产是商品数据，不是订单系统。
2. 先把“会推荐、会比较、会找相似款”做强，最容易看到价值。
3. 交易工具后补，不会阻塞导购核心能力验证。

## 15. 结论

你的方向是对的，而且和这 5 个近期项目的共识高度一致：

1. 用商品知识做 RAG
2. 用图片和文本做多模态召回
3. 用工具调用完成加购、下单、取消订单、退货
4. 用状态机和权限控制约束交易动作
5. 用主 Agent + subagent 组织复杂购物任务

真正决定成败的关键，不是“有没有 Agent”，而是下面这三件事：

1. 商品数据清洗是否扎实
2. 检索与排序是否稳定
3. 交易与售后工具是否可靠可控

如果按这个路线推进，`MallPilot` 很适合先做成一个“会推荐、会比较、会加购、会售后协助”的导购 Agent，再逐步升级成完整的电商购物助手。
